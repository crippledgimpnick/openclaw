# SOP Roadmap

## Purpose
This document lists the standard operating procedures that should be created as the GoHighLevel purchasing system matures.

The system should not rely only on automation. Humans need a repeatable playbook.

## SOP Priority Order
1. New Lead Review SOP
2. Qualification SOP
3. Quote Review SOP
4. Negotiation Escalation SOP
5. Shipment Tracking SOP
6. Receipt and Inspection SOP
7. Payment Release SOP
8. Repeat Seller SOP
9. Reactivation SOP
10. Issue / Rollback SOP

---

## 1. New Lead Review SOP
Should define:
- how often new leads are checked
- what counts as a valid new lead
- how owner assignment is handled
- what to do with incomplete source records
- response speed expectations

## 2. Qualification SOP
Should define:
- required information
- acceptable photo standards
- what to do with partial responses
- when to trigger manual review
- when a deal can move to quote review

## 3. Quote Review SOP
Should define:
- who approves quotes
- what data must be present before approval
- how exceptions are handled
- what gets documented before a quote is sent

## 4. Negotiation Escalation SOP
Should define:
- when the workflow can continue automatically
- when human review is required
- what authority limits exist
- what language must be avoided

## 5. Shipment Tracking SOP
Should define:
- what instructions are sent after acceptance
- how tracking is logged
- how long to wait before shipment reminders
- what to do if the seller says they shipped but no tracking is available

## 6. Receipt and Inspection SOP
Should define:
- how receipt is confirmed
- how condition and quantity are checked
- how discrepancies are handled
- who can approve for payment

## 7. Payment Release SOP
Should define:
- approved payment methods
- payment timing expectations
- what triggers payment pending vs payment complete
- what documentation is required before payment release

## 8. Repeat Seller SOP
Should define:
- how repeat sellers are identified
- whether they get different prioritization
- whether they get different follow-up handling
- how history affects trust and manual review

## 9. Reactivation SOP
Should define:
- which stale leads are worth re-engaging
- timing windows
- message frequency limits
- when to stop trying

## 10. Issue / Rollback SOP
Should define:
- how bad workflow changes are identified
- what gets rolled back first
- who reviews automation failures
- how incidents are logged in the changelog

## Notes
These SOPs should be created after the core system structure is built, but before the system is considered mature.
